§ 4 · 4 décisions humaines qui t'attendent

De la plus critique à la plus optionnelle.

Ces 4 décisions ne pouvaient pas être prises automatiquement (jugement business, action externe, ou risque de casser quelque chose). À toi de trancher avant le launch jeudi 7 mai.

D1

Valider la capture Sentry en prod (5 min)

5 min

Hier soir on a déployé le SDK Sentry + posé le DSN sur Railway. L'endpoint debug /api/health/sentry-test est en place mais on n'a pas validé que les erreurs arrivent bien dans le dashboard Sentry. Test en 1 commande, puis on supprime l'endpoint debug.

# 1. Lance ce curl (token déjà set sur Railway)
curl -s -H "x-sentry-test: 62qgxFee5wp04ME9lmkptw" \
  https://www.openswissdata.com/api/health/sentry-test

# 2. Va sur openswissdata.sentry.io → Issues → Feed
# 3. Tu dois voir une nouvelle erreur "Sentry smoke test — intentional throw"
# 4. Si OK, dis-moi → je supprime l'endpoint debug + le token Railway
Pourquoi prioritaire : Sentry sera le seul moyen de comprendre les erreurs runtime quand tu auras des vrais clients. Si la capture est cassée silencieusement, tu perds toute visibilité. 5 min pour valider une fois pour toutes.
D2

Self-purchase LIVE test sur les 4 SKUs (~40 min)

~40 min

Stripe LIVE est actif depuis le 25/04 mais aucun vrai paiement n'a jamais traversé le webhook signé en LIVE. Le E2E complet (Stripe Live → webhook → entitlement DB → email Resend → magic link → download R2) doit être validé avant le 1er client réel. Coût : ~30 CHF de fees Stripe (refundés).

A. Faire le test maintenant Procédure complète dans .design-preview/sentry-and-self-purchase-guide.html § Action 6. 4 SKUs × ~10 min : TARES (299), Classifications (399), FINMA (299), Bundle (799).
B. Reporter à mercredi 6 mai Risque : si un bug est découvert mercredi, peu de temps pour fixer avant le launch jeudi.
Pourquoi prioritaire : aucun launch sans ce test n'est sain. Sentry + UptimeRobot vont aussi être validés simultanément (chaque erreur sera capturée, chaque check sera vert).
D3

Bundle Stripe : 799 CHF ou 797 CHF ?

~5 min côté Stripe

Audit UX a flaggé que 997 − 799 = 198 CHF (19.86 %) alors que les copys disaient "200 CHF (20 %)" partout. J'ai patché tous les copys avec "≈ 200 CHF (≈ 20 %)" — math honnête mais avec tildes. Décision business : préfères-tu une math parfaite ou le prix actuel ?

A. Garder 799 CHF (actuel) Aucune action, les copys avec ≈ sont déjà déployés. Laisse-toi marge fiscale optimale.
B. Passer à 797 CHF Stripe Dashboard → Products → Bundle → Edit price → 797 CHF. Recréer le webhook? Non, juste le prix. Math nickel : 997 − 797 = 200 CHF (20.06 %). Je remets les copys sans tilde après ton OK.
Pourquoi non-bloquant : avec les tildes les copys sont honnêtes et défendables. Mais "200 CHF · 20 %" pile c'est plus propre marketing-wise. Différence revenue : 2 CHF par bundle vendu, soit ~100 CHF/an pour 50 bundles. Négligeable.
D4

Contraste --ink-mute WCAG AA (5 min)

5 min

Audit a11y a identifié que --ink-mute (#8A8D95) sur --bg (#FBFBF8) donne ratio 3.2:1 → échec WCAG AA (seuil 4.5:1 pour body 16px). Utilisé dans les sous-titres mono, les captions, les meta. Je n'ai pas appliqué automatiquement car ça impacte le branding : le gris devient plus foncé partout.

A. Passer à #6B6E76 Ratio 5.0:1 → conforme WCAG AA. Nuance plus foncée mais rest dans la même famille gris-bleu. Compatible EU Accessibility Act juin 2025 obligatoire pour B2B vers UE.
B. Garder #8A8D95 Branding actuel préservé. Risque : non-conformité légale si client EU teste avec axe / WAVE / Lighthouse.
Comment décider : regarde une page comme /datasets/tares avec les deux variantes (mute clair vs foncé) et juge visuellement. C'est subjectif. Si tu hésites, prends A (conformité légale > 0.5 % de fade aesthétique).
