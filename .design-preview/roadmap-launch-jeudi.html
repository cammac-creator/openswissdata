Phase 1 / Critique / 3 chantiers à finir avant mercredi soir DEADLINE MERCREDI 6 MAI

Les 3 actions qui débloquent le launch.

Sans ces 3 chantiers, tu peux pas annoncer "premier MCP suisse / 8 tools" sans risquer une plainte ou un bug client le jour J. À régler dans cet ordre.

01
Bundle Stripe : passer 799 → 797 CHF
Critique À faire toi ~5 min
Ton site affiche partout "≈ 200 CHF économisés (≈ 20 %)". La math n'est exacte que si le bundle coûte 797 CHF (299 + 399 + 299 = 997 → 997 − 797 = 200). À 799 CHF, l'écart est de 198 CHF — pas faux mais pas net.
Pourquoi ça compte : un acheteur compliance qui calcule à la main remarque tout de suite. Fais le geste maintenant pour que tes pages disent exactement ce que tu factures, sans le ≈.
Maintenant (Stripe)
799 CHF
997 − 799 = 198 CHF (≈ 20 %)
Cible
797 CHF
997 − 797 = 200 CHF tout rond ✓
Comment faire
  1. Aller sur dashboard.stripe.com/products
  2. Trouver le produit "Bundle openswissdata"
  3. Cliquer sur le prix LIVE actuel (799 CHF)
  4. "Update price" → entrer 797.00 CHF
  5. Sauvegarder, copier le nouveau price_id
  6. Me le coller dans le chat → je mets à jour le code et je redéploie
02
Test OAuth E2E des 5 tools Pro MCP
Critique Je peux faire ~30 min
Ton MCP (Model Context Protocol) expose 8 tools : 3 publics anonymes (déjà validés en prod) + 5 Pro qui exigent une authentification OAuth + scope. Les 5 Pro n'ont jamais été testés en condition réelle de bout en bout.
Pourquoi ça compte : tu prévois de communiquer "premier MCP suisse, 8 tools". Si un journaliste tech ou un dev influent essaie un tool Pro et que ça plante (token mal scoped, rate limit, JSON malformé), tu perds la crédibilité du launch. Mieux vaut le savoir maintenant.
Comment faire (je peux le faire pour toi)
  1. Je génère un token OAuth Pro de test
  2. Je lance un script qui appelle chacun des 5 tools (tariff_lookup_pro, tariff_changelog, etc.)
  3. Je vérifie : 200 OK + JSON conforme au schéma + temps de réponse < 1s
  4. Je note les bugs (s'il y en a) et je les fixe
  5. Tu auras un rapport "5/5 OK" ou "4/5 OK + bug fix"

→ Dis-moi "go test mcp" et je lance.

03
Contraste --ink-mute (a11y WCAG AA)
Critique a11y Décision visuelle ~10 min
La couleur de tes textes "secondaires" (sous-titres, légendes, breadcrumbs) est --ink-mute: #8A8D95 sur fond ivoire #FBFBF8. Le ratio de contraste est 3.2:1 — en dessous du seuil WCAG AA (4.5:1). C'est un bug d'accessibilité.
Pourquoi ça compte : 1 visiteur sur 8 a une vue altérée (myopes corrigés, malvoyants, daltonisme). Les textes en gris clair leur sont illisibles. C'est aussi un argument compliance que tes clients banques/assurances vont vérifier (audit a11y obligatoire en CH dès 2025 pour le public).
Actuel
Texte secondaire (3.2:1)
#8A8D95 · échec WCAG AA
Cible
Texte secondaire (4.6:1)
#6B6E76 · WCAG AA ✓
Comment faire
  1. Tu valides visuellement la cible #6B6E76 (peu de différence à l'œil nu, juste un poil plus foncé)
  2. Tu me dis "go contraste" → je modifie web/src/styles/global.css
  3. Je redéploie, tu vérifies en live sur 2-3 pages que ça reste lisible et beau
Phase 2 / Optimisation / 4 chantiers — peuvent attendre après launch

Les 4 actions "nice to have".

Pas bloquantes pour jeudi, mais à régler dans les 2 semaines suivantes pour que la base soit solide. Aucune n'est visible côté client.

04
Refactor /codes : Astro Island lazy-load
Performance Je peux faire ~2 h
La page /codes/index.html pèse 396 KB à cause de 1047 codes NOGA inlinés en HTML statique. C'est lourd pour mobile 3G/4G, et pénalise le Core Web Vital LCP.
Pourquoi ça compte : Google indexe cette page pour 1047 mots-clés SEO (un par code NOGA). Si elle est lente, les positions SEO baissent, et tu perds des leads organiques.
Comment faire
  • Garder seulement le top 50 codes en HTML statique (les plus recherchés)
  • Les 1000 autres dans un JSON chargé à la demande quand l'user scrolle ou cherche
  • Cible : passer de 396 KB à < 100 KB
05
Bug scope MCP : tariff_changelog → tariff:read
Bug mineur Je peux faire ~15 min
Le tool MCP tariff_changelog est mappé sur le scope tariff:semantic alors qu'il devrait être sur tariff:read (ou créer un nouveau scope tariff:history). C'est un bug identifié pendant l'audit MCP E2E.
Pourquoi ça compte : certains clients qui achètent l'accès "lecture" (tariff:read) ne pourront pas utiliser changelog, alors qu'ils devraient. Friction commerciale.
Comment faire
  1. Modifier le mapping dans sdks/mcp-server/src/tools.ts
  2. Vérifier les tests unitaires
  3. Push live
06
Supprimer Header.astro (code mort)
Cleanup Je peux faire ~5 min
Le composant web/src/components/Header.astro existe dans le repo mais n'est importé nulle part — c'est StatusBar.astro qui est utilisé en prod. Le fichier prend de la place et embrouille.
Pourquoi ça compte : du code mort dans un repo, c'est de la dette qui grossit. Quelqu'un (toi, moi, ou un futur dev) finira par l'éditer par erreur ou perdre 10 min à comprendre s'il sert.
Comment faire
git rm web/src/components/Header.astro
git commit -m "chore: remove unused Header.astro (StatusBar is the active component)"
git push
07
Validation Sentry capture en prod
Observabilité À faire toi ~5 min
Sentry est déployé sur Railway, le DSN est configuré. Mais on n'a jamais vérifié qu'une erreur API remonte bien dans le dashboard Sentry.
Pourquoi ça compte : le jour du launch, si une erreur 500 frappe, tu dois la voir en temps réel pour réagir. Sentry est ton radar. Si le radar est éteint, tu ne sais pas ce qui se passe.
Comment faire
  1. Lance curl -s -H "x-sentry-test: 62qgxFee5wp04ME9lmkptw" https://www.openswissdata.com/api/health/sentry-test
  2. Ouvre sentry.io → projet openswissdata → Issues
  3. L'erreur de test doit apparaître dans les 30 secondes
  4. Si elle apparaît → ✅ Sentry est vivant, je supprime l'endpoint debug
  5. Si elle n'apparaît pas → on diagnostique ensemble
